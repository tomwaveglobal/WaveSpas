## Stream 1.2 — Review schema audit + fix

### Finding 1: There is NO Product, BreadcrumbList, FAQPage, or aggregateRating JSON-LD on the site

Confirmed by inspecting `layout/theme.liquid` and `sections/main-product.liquid`:
- No `<script type="application/ld+json">` blocks in the theme
- Shopify auto-injects *some* Product schema via `{{ content_for_header }}`, but it does NOT include aggregateRating from your review data
- The 31k Review snippet impressions in your GSC data are likely from Google extracting data from the Trustpilot image alt text or page text — not from proper schema

### Finding 2: Reviews come from Reviews.co.uk + a possible bug

In [snippets/product-rating.liquid](snippets/product-rating.liquid):

```liquid
{%- if product.metafields.reviews.rating.value != blank and product.metafields.reviews.rating_count.value > 0 -%}
  ...
  {{ product.metafields.reviewscouk.rating.value }}
  ...
```

The conditional checks `product.metafields.reviews` (the native Shopify Reviews namespace) but the actual rating value displayed is from `product.metafields.reviewscouk` (the Reviews.co.uk app namespace).

This will work IF the Reviews.co.uk app syncs to both namespaces. Worth verifying — if it doesn't, ratings won't render on PDPs where data only exists in one namespace.

### Finding 3: Trustpilot is shown as an image, not a widget

[products/garda-hot-tub](https://wavespas.com/products/garda-hot-tub) shows "Trustpilot_Spas.png" — a marketing image, not an embedded Trustpilot TrustBox widget. So:
- Trustpilot review data is NOT in the page DOM
- Google cannot extract aggregateRating from Trustpilot
- Only Reviews.co.uk data (if it's in the page) could feed review snippets

### The fix — 3 actions

**Action A — Add Product JSON-LD with aggregateRating to PDPs** (code change)

In [sections/main-product.liquid](sections/main-product.liquid), add this block immediately before the closing `{% schema %}` tag:

```liquid
{%- if product != blank -%}
  <script type="application/ld+json">
  {
    "@context": "https://schema.org/",
    "@type": "Product",
    "name": {{ product.title | json }},
    "description": {{ product.description | strip_html | truncate: 5000 | json }},
    "image": [
      {%- for image in product.images limit: 5 -%}
        {{ image | image_url: width: 1200 | prepend: 'https:' | json }}{%- unless forloop.last -%},{%- endunless -%}
      {%- endfor -%}
    ],
    "sku": {{ product.selected_or_first_available_variant.sku | json }},
    "brand": {
      "@type": "Brand",
      "name": "Wave Spas"
    },
    "offers": {
      "@type": "Offer",
      "url": {{ shop.url | append: product.url | json }},
      "priceCurrency": {{ cart.currency.iso_code | json }},
      "price": {{ product.selected_or_first_available_variant.price | money_without_currency | strip_html | remove: ',' | json }},
      "availability": "{% if product.available %}https://schema.org/InStock{% else %}https://schema.org/OutOfStock{% endif %}",
      "itemCondition": "https://schema.org/NewCondition"
    }
    {%- if product.metafields.reviewscouk.rating.value != blank and product.metafields.reviewscouk.rating_count.value > 0 -%}
    ,"aggregateRating": {
      "@type": "AggregateRating",
      "ratingValue": {{ product.metafields.reviewscouk.rating.value | json }},
      "reviewCount": {{ product.metafields.reviewscouk.rating_count.value | json }},
      "bestRating": "5",
      "worstRating": "1"
    }
    {%- endif -%}
  }
  </script>
{%- endif -%}
```

**What this does:**
- Adds explicit Product schema even when Shopify's default is present (the explicit one is more reliable)
- Pulls aggregateRating from Reviews.co.uk metafields (which is your actual review source)
- Adds the Offer block with proper price/availability
- Conditional on aggregateRating data existing so it doesn't fire empty

**Forecast:** Review rich snippets fire on PDPs where reviews exist. CTR uplift on product snippet impressions: +10–20% (industry benchmark for adding star ratings to SERP listings). With 3.6M product snippet impressions/16mo, even a 5% relative CTR lift = **+2,800 clicks/mo**.

**Action B — Fix the metafield namespace mismatch in product-rating.liquid**

In [snippets/product-rating.liquid](snippets/product-rating.liquid), change line 1 of the condition:

```liquid
{%- if product.metafields.reviews.rating.value != blank and product.metafields.reviews.rating_count.value > 0 -%}
```

to:

```liquid
{%- if product.metafields.reviewscouk.rating.value != blank and product.metafields.reviewscouk.rating_count.value > 0 -%}
```

This ensures the rating stars actually render based on the Reviews.co.uk data the rest of the snippet uses. (Verify by checking if a product known to have reviews currently shows stars — if it does, the namespace mismatch is being masked by both metafields being populated.)

**Action C — Replace Trustpilot image with real Trustpilot widget (optional)**

If you want Trustpilot reviews to also feed Google rich snippets, replace the "Trustpilot_Spas.png" image with a real Trustpilot TrustBox widget (Trustpilot dashboard → Showcase → TrustBoxes → choose Mini Carousel or Product Review). This requires:
1. Trustpilot Business account
2. Connecting product reviews to Shopify SKUs

Worth doing for the longer term but **Action A above is the higher-priority fix** since it works with your existing Reviews.co.uk data immediately.

## Stream 1.3 — Desktop ranking diagnostic

### The problem
Mobile rank 10.27 vs desktop rank 21.58 — desktop is 2x worse than mobile. That's not normal device-CTR variance; it's a systemic ranking gap.

### My hypotheses (ranked by likelihood)

#### H1: Desktop LCP is the hero video / large carousel image (MOST LIKELY)
Evidence:
- Live homepage uses video content + carousel images (confirmed via WebFetch)
- Mobile likely loads a smaller image variant; desktop loads the video preview/full-res image
- Hero media has no explicit `width`/`height` attributes set in [layout/theme.liquid](layout/theme.liquid) header → CLS risk
- Trustpilot image is loaded as a regular `<img>`, which can cause layout shift

**Why this matters:** Google uses Core Web Vitals as a ranking signal, measured separately for mobile and desktop. Desktop CWV failing → desktop pages get ranked lower.

#### H2: Mobile-first design has under-optimised desktop layouts
Evidence from theme code:
- Heavy `lg:` and `md:` Tailwind responsive classes throughout
- Patterns like `hidden md:block` mean content gets hidden conditionally
- Desktop layout breakpoints (1024px+) often added as afterthought

#### H3: Reviews.co.uk widget / scripts blocking desktop render path
Not confirmed yet — needs Lighthouse audit.

### Investigation steps Tom should run (15 minutes)

**Step 1:** Run **PageSpeed Insights** on these 3 URLs, both Mobile and Desktop:
- https://pagespeed.web.dev/analysis?url=https%3A%2F%2Fwavespas.com%2F
- https://pagespeed.web.dev/analysis?url=https%3A%2F%2Fwavespas.com%2Fcollections%2Finflatable-hot-tubs
- https://pagespeed.web.dev/analysis?url=https%3A%2F%2Fwavespas.com%2Fproducts%2Fgarda-hot-tub

Note for each: **LCP score, CLS score, total CWV pass/fail**. Compare mobile vs desktop.

**Step 2:** Check **GSC → Experience → Core Web Vitals** report. Filter by Mobile and Desktop separately. If desktop has a much higher "needs improvement" or "poor" URL count → confirmed.

**Step 3:** Drop the Lighthouse scores into chat (or screenshot the GSC CWV report). I'll prescribe specific fixes based on what fails.

### Likely fixes (predictions before data arrives)

If H1 is right (desktop LCP problem):
- Add explicit `width` and `height` to hero media in image-with-text-overlay sections
- Use `<link rel="preload">` for the desktop LCP image specifically
- Convert hero videos to image+poster on desktop (Shopify videos are heavy)
- Defer Trustpilot image loading via `loading="lazy"`

If H2 is right (mobile-first under-optimised):
- Add desktop-specific image sizes (currently most components only specify mobile sizes)
- Audit `lg:hidden`/`hidden md:block` patterns for unintended content hiding

**Forecast (if root cause is in this list):** Desktop hitting mobile parity = **+3,400 clicks/mo**.
